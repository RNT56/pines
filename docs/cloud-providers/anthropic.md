## What Pines Supports Today

- Anthropic Messages streaming with BYOK credentials.
- Model listing from `/v1/models` and validation through `/v1/messages`.
- System prompt mapping to the `system` field.
- Image, PDF, and UTF-8 text-document inputs on user messages.
- Function tool definitions and tool result handling.
- Native web search through `web_search_20250305` with domain filters and approximate location.
- Adaptive thinking for recognized Claude models, with hidden thinking capture and replay.
- Anthropic `output_config.effort` for recognized models.
- Basic prompt-caching intent when chats contain enough context, tools, attachments, or system messages.
- Request ID and token usage metadata parsing.

## High-Value Unsupported Or Partial Features

### 1. Files API and file ID reuse

Pines inlines documents/images today and does not upload or reference Anthropic `file_id`s.

Value:

- Avoid repeated upload cost and latency for large PDFs, images, datasets, and code-execution inputs.
- Enables files generated by Anthropic code execution to be retrieved.

Implementation notes:

- Add file upload/list/delete/download lifecycle with the `files-api-2025-04-14` beta header.
- Support document/image blocks that reference `file_id`.
- Treat file persistence as provider-hosted storage distinct from local Vault.

### 2. Citations for source-grounded documents

Pines sends PDFs/text but does not request or parse Anthropic citation blocks.

Value:

- Verifiable answers over user documents.
- Better UX for legal, finance, research, and technical reading.

Implementation notes:

- Add citation-enabled document blocks for plain text, PDF, and custom content.
- Parse citation metadata into `WebSearchCitation`-like source references.
- Convert unsupported file types to plain text or route them through code execution/files.

### 3. Correct and explicit prompt caching

Anthropic prompt caching is block-oriented (`cache_control`) with TTL and minimum-token rules. Pines currently sets a top-level `cache_control` object, which should be audited against current Anthropic request requirements.

Value:

- Major cost and latency reductions for long system prompts, tools, and repeated vault context.

Implementation notes:

- Put `cache_control` on the right system/tool/message content blocks.
- Expose 5-minute versus 1-hour TTL where supported.
- Parse cache creation/read usage fields and surface savings.

### 4. Extended thinking controls

Pines uses adaptive thinking for recognized model IDs, but it does not expose full thinking budgets, summaries, or task-budget controls.

Value:

- Users can trade cost/latency for quality on hard reasoning tasks.
- Better parity with Claude's advanced reasoning modes.

Implementation notes:

- Add per-model thinking mode controls: off, adaptive, explicit budget, and provider-supported effort.
- Decide how much thinking summary, if any, should be displayed.
- Preserve thinking signatures correctly across tool-result turns.

### 5. Code execution tool

Anthropic now exposes hosted code execution with Bash and file operations on supported Claude models. Pines does not include `code_execution_20250825` or newer tool versions.

Value:

- Data analysis, file transformation, charts, calculations, code execution, and document processing inside Claude.

Implementation notes:

- Requires beta/tool version handling.
- Parse `server_tool_use`, `bash_code_execution_tool_result`, and `text_editor_code_execution_tool_result`.
- Add generated-file retrieval through Files API.
- Keep a clear UI distinction between Anthropic's sandbox and local tools.

### 6. Web fetch tool

Pines supports Anthropic web search but not web fetch.

Value:

- Fetch and inspect specific URLs rather than search the web broadly.
- Useful for user-provided links, documentation, articles, and issue pages.

Implementation notes:

- Add provider-level URL approval and source display.
- Consider pairing with code execution, since Anthropic can process fetched content before it reaches context.

### 7. Remote MCP connector

Pines has local MCP support but does not pass Anthropic `mcp_servers` to Messages.

Value:

- Let Claude use remote MCP tools without Pines acting as the MCP client.
- Useful for teams with hosted MCP servers and OAuth bearer tokens.

Implementation notes:

- Store remote MCP server configs, tokens, and allowed-tool lists.
- Differentiate provider-hosted MCP calls from Pines-local MCP calls in audit logs and consent UI.

### 8. Computer use, bash, text editor, advisor, memory, and tool search

Anthropic exposes several provider-side tools that Pines does not integrate.

Value:

- Computer use: browser/desktop automation from screenshots and actions.
- Bash/text editor: coding and file manipulation workflows.
- Memory/advisor/tool search: better long-running agents and large tool surfaces.

Implementation notes:

- Most should be agent-only with approval gates.
- Parse provider-specific tool result block types.
- Avoid overlapping local and provider tools without explicit environment labels.

### 9. Fine-grained tool streaming and parallel tool use controls

Pines parses normal tool deltas but does not expose Anthropic's fine-grained tool streaming or detailed parallel tool behavior controls.

Value:

- Lower latency for large tool arguments and better multi-action agents.

Implementation notes:

- Add beta header and parser coverage for partial parameters.
- Revisit Pines' single-tool-call execution loop and approval UX.

### 10. Message Batches API

Pines does not use Anthropic batch processing.

Value:

- Lower-cost asynchronous bulk summarization, classification, document extraction, and vault maintenance.

Implementation notes:

- Add batch job records, retry/resume, result import, and cost/latency warnings.

### 11. Token counting and context management APIs

Pines does not call Anthropic token counting before sending requests.

Value:

- Better preflight errors, attachment sizing, vault context packing, and cost estimates.

Implementation notes:

- Use provider token count APIs for request budgeting.
- Integrate with `ExecutionRouter` and Vault context selection.

### 12. Structured outputs and search-result blocks

Pines uses free-form text and function tools, but not Anthropic structured output guidance or `search_result` blocks for custom RAG citations.

Value:

- Schema-reliable extraction.
- Citation-quality custom retrieval without using Anthropic web search.

Implementation notes:

- Add provider-neutral schema requests where possible.
- Map Pines Vault search snippets into Anthropic search-result blocks when user consents to cloud context.

## Suggested Priority

1. Fix/complete prompt caching.
2. Files API and citation parsing.
3. Code execution and generated-file retrieval.
4. Extended thinking controls.
5. Web fetch and remote MCP.
6. Token counting and batch processing.
7. Computer use/text editor/bash/memory/tool search.

## Review Checklist

- Should Anthropic hosted files be a cloud mirror of Vault documents or a separate file library?
- Which provider tools are safe in normal chat, and which must be agent-only?
- Should Anthropic citations become the first cloud citation model for Vault context?
- Do we want user-facing thinking summaries, hidden thinking preservation only, or both?
- Should provider-hosted MCP be enabled when Pines already supports local MCP?
